4) Target -> Create all necessary tablespaces with same specification as source database. 
5) Target -> Enable backups for target database through 'enable database protection' under backup & cloning tab on database page at DST portal. 
			 Please ensure backups are running before starting import. Don't enable backups for DEV databases unless needed.
			 
  By default archives will be retained on disk for 60 minutes. Import may generate too many archives and disk space may reach to 100% by the time backup runs.
  You can decrease backup retention to less value using below procedure. 
  
  exec sys.oraas_util.update_rman_archive_retention(30,'Y');
  
  select * from sys.CONFIG_RMAN_ARCHIVE_DELETION;
  
  Don't forget to revert it back to 60 minutes after the completion of import.
  
  exec sys.oraas_util.update_rman_archive_retention(60,'Y');
  
  --5 hours
  exec sys.oraas_util.update_rman_archive_retention(300,'Y');

  --24 hours
  exec sys.oraas_util.update_rman_archive_retention(1440,'Y');
  
    exec sys.oraas_util.update_rman_archive_retention(300,'N');


  
  Databases that don't have backup protection will have 'delete archivelogs' operation in case space is 100% used and import/DB is stuck

	This operation may fail if there is no enough space on DATA_PUMP_DG diskgroup on OraaS GRID.
	You can check the total usage and usage by database by selecting 'DATA_PUMP_DG Total Usage' and 'DATA_PUMP_DG Usage by database' options under commands tab on database details page.
	You can reach out to PDBA of the database that uses majority of the space and see if the dump files can be cleaned.  
		
	Also clean old dump of your database by selecting change and 'Cleanup DataPumpDG' from operations tab on database details page.
	If more space is needed than total size of DATA_PUMP_DG try using compression option in datapump export.
	If that is not feasible reach out to OraaS PROD support/L3 team to get the space increased.	
	
	12c clusters have an ACFS filesystem /datapumpacfs instead of +DATA_PUMP_DG diskgroup.
	Below procedures can be used to list and clean contents of ACFS directories specific to the database.
	
	PROCEDURE LIST_ACFS_DIR(F_STR IN VARCHAR2 := NULL)
	
	This procedure is used to list the files of ACFS filesystem database directory, 
	if no file string parameter passed, it displays all the files, if file string parameter passed, 
	it only display the files with file string.
	examples of the execution of this proc is:
	SQL> set serveroutput on
	SQL> exec sys.oraas_util.LIST_ACFS_DIR;
	SQL> exec sys.oraas_util.LIST_ACFS_DIR('dmp');
	here "set serveroutput on" is required since it is calling dbms_ouput_put_line to display the files.
	
	
	PROCEDURE CLEAN_ACFS_DIR(F_STR IN VARCHAR2 := NULL)
	
	This procedure is used to remove the files under ACFS filesystem database directory, 
	if no file string parameter passed, it cleans all the files, if file string parameter passed, 
	it only clean the files with file string.
	examples of the execution of this proc is:
	SQL> exec sys.oraas_util.clean_ACFS_DIR;
	or
	SQL> exec sys.oraas_util.clean_ACFS_DIR('DMP');


	If you still have space issue for transferring dumps, work with OraaS team, find out databases consuming /datapumpacfs.
	Reach out PDBA/SDBA of those databases asking for cleanup
	
	If source database is huge, consider using compress=y option in export

8) Target --> Verify if files are successfully transferred using below query. You can crosscheck the size of dump files.

set pages 100
set lines 300
col full_path for a80
col permissions for a12
col creation_date for a20
col bytes for 999,999,999,999,999

select concat('+'||gname, sys_connect_by_path(aname, '/')) full_path,
         bytes,
         permissions,
         creation_date
    from ( select b.name gname, a.parent_index pindex, a.name aname,
              a.reference_index rindex , a.system_created, a.alias_directory, c.bytes, c.PERMISSIONS, c.CREATION_DATE,
              c.type file_type
       from v$asm_alias a, v$asm_diskgroup_stat b, v$asm_file c
       where a.group_number = b.group_number
             and b.NAME='DATA_PUMP_DG'
             and a.group_number = c.group_number(+)
             and a.file_number = c.file_number(+)
             and a.file_incarnation = c.incarnation(+)
      )
    where system_created = 'N'
    start with (mod(pindex, power(2, 24))) = 0
            and rindex in
                ( select a.reference_index
                  from v$asm_alias a, v$asm_diskgroup_stat b, v$database d
                  where a.group_number = b.group_number
                    and (mod(a.parent_index, power(2, 24))) = 0
                    and a.name = d.name
                )
    connect by prior rindex = pindex
	order by full_path;

	
select distinct full_path, bytes, creation_date, type from 
(
SELECT CONCAT('+' || db_files.disk_group_name,
	  SYS_CONNECT_BY_PATH(db_files.alias_name, '/')) full_path ,
      round(db_files.bytes/1024) bytes ,
	  db_files.creation_date ,
	  NVL(LPAD(db_files.type, 18), '') type 
	  FROM 
	  ( SELECT 
		g.name disk_group_name ,
		a.parent_index pindex ,
		a.name alias_name ,
		a.reference_index rindex ,
		a.system_created system_created ,
		f.bytes bytes , f.type type ,
		TO_CHAR(f.creation_date, 'DD-MON-YYYY HH24:MI:SS') creation_date 
		FROM 
		v$asm_file f RIGHT OUTER JOIN v$asm_alias a 
		USING (group_number, file_number) JOIN v$asm_diskgroup g 
		USING (group_number) where g.name = 'DATA_PUMP_DG' ) db_files 
		WHERE db_files.type IS NOT NULL and db_files.system_created = 'N' 
		START WITH (MOD(db_files.pindex, POWER(2, 24))) = 0 
		CONNECT BY PRIOR db_files.rindex = db_files.pindex ) 
		where upper(full_path) like '%/'||sys_context('userenv','db_unique_name')||'/%';
